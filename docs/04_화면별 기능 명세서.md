# 화면별 기능 명세서

## 💡 업무 기획

### 업무 목적

1. **바이브 코딩 시 해석 차이**
    1. ‘URL 입력 수집’이라고만 써 있으면, 개발자가 단일 URL만 지원할지, 복수 URL을 지원할지, 유효성 검증은 어떻게 할지 각자 판단하게 됨
2. **디자인 누락**
    1. ‘수집 실패 시 어떻게 보여주지?’가 정의 안 되어 있으면 프로토타입에서 에러 화면이 빠짐
3. **킥오프 이후 커뮤니케이션 비용 폭증**
    1. 온꿈사 팀과 협업할 때 ‘이 버튼 누르면 뭐가 일어나요?’를 매번 구두로 설명해야 함

### 화면 1개당 작성 구조

```
[화면 ID] 화면명
├── 1. 화면 목적 (이 화면이 왜 존재하는가)
├── 2. 진입 조건 (어떤 상태에서 이 화면에 도달하는가)
├── 3. UI 요소별 기능 정의 (테이블)
│   ├── 요소명
│   ├── 유형 (입력/표시/버튼/토글 등)
│   ├── 동작 설명 (사용자가 X하면 시스템이 Y한다)
│   ├── 비즈니스 룰 (조건/제약/계산식)
│   └── 에러/예외 처리
├── 4. 상태 변화 (이 화면에서 데이터 상태가 어떻게 바뀌는가)
├── 5. API/데이터 의존성 (어떤 API를 호출하는가)
└── 6. 이탈 경로 (이 화면에서 어디로 갈 수 있는가)
```

### 0. 공통 (Global)

<aside>
💡

**설계 의도**: 모든 화면에 공통으로 노출되는 글로벌 UI 요소. GNB는 셀러의 파이프라인 단계를 바로 인지할 수 있는 내비게이션, 시스템 알림은 백그라운드 작업 결과를 셀러에게 전달하는 유일한 채널.

</aside>

#### [G-01] GNB (글로벌 네비게이션)

**1. 화면 목적**

모든 화면 상단에 고정되어 셀러가 파이프라인 단계(소싱→편집→등록→내 상품) 간 전환과 시스템 상태 확인을 할 수 있는 주 내비게이션.

**2. 진입 조건**

- 로그인 완료 상태여야 표시
- 모든 화면에서 상단 고정 노출 (`S-01 로그인/회원가입` 제외)

**3. UI 요소별 기능 정의**

| **요소명** | **유형** | **동작 설명** | **비즈니스 룰** | **에러/예외** |
| --- | --- | --- | --- | --- |
| **로고** | 버튼 | 클릭 시 로그인 후 기본 랜딩(`SC-01 소싱 홈`)으로 이동 | 어떤 화면에서든 동일 동작 | — |
| **메인 메뉴** | 탭 내비게이션 | 탭 클릭 시 해당 섹션 메인 화면으로 이동. **현재 활성 섹션**은 하이라이트 표시(색상/언더라인) | 탭 순서: 소싱 · 편집 · 등록 · 내 상품 · 설정 (파이프라인 순서 반영) | — |
| **알림 아이콘** | 아이콘 버튼 | 클릭 시 `G-02 시스템 알림` 패널 오픈. 아이콘 우측 상단에 **미독 수 뼉지** 표시 | 미독 알림 0건 → 뼉지 숨김. 99건 초과 → "99+" 표시 | 알림 로드 실패 시 아이콘만 표시, 뼉지 숨김 |
| **사용자 프로필** | 아바타 버튼 | 클릭 시 드롭다운 메뉴 표시: `ST-01 계정 관리` · 로그아웃 | 프로필 이미지 미설정 시 이니셜 아바타 자동 생성 | — |
| **파이프라인 진행 상태 표시** | 표시(선택적) | 각 메뉴 탭 우측에 해당 단계의 처리 건수 표시 (ex. 편집 "3", 등록 "확인") | 소싱: 미편집 상품 수 / 편집: AI완료+미편집 수 / 등록: 확인 필요 수 / 내 상품: 조치필요 수 | 건수 0 → 숨김 (깨끗한 상태 유지) |

**4. 상태 변화**

- GNB 자체는 데이터를 변경하지 않음 (순수 네비게이션)
- 알림 뼉지 수는 `G-02 시스템 알림` 패널 오픈 시 "0"으로 초기화되지 않음 — 개별 알림 클릭 시 해당 건만 읽음 처리

**5. API/데이터 의존성**

- 내부 알림 DB (`G-02 시스템 알림` 미독 수 조회)
- 각 파이프라인 단계별 건수 집계 API (소싱 DB, 편집 DB, 등록 결과 DB, 변동 확인 DB)

**6. 이탈 경로**

- `SC-01 소싱 홈`: "소싱" 탭
- `ED-01 편집 목록`: "편집" 탭
- `RG-02 등록 결과`: "등록" 탭
- `PM-01 등록 상품 목록`: "내 상품" 탭
- `ST-01 계정 관리`: "설정" 탭 또는 프로필 드롭다운
- `G-02 시스템 알림`: 알림 아이콘 클릭

---

#### [G-02] 시스템 알림

**1. 화면 목적**

백그라운드에서 자동 수행된 작업 결과(수집 완료, 등록 성공/실패, 변동 확인/역마진)를 셀러에게 전달하고, 해당 화면으로 빠르게 이동시키는 **중앙 알림 허브**.

**2. 진입 조건**

- `G-01 GNB` 알림 아이콘 클릭 시 사이드 패널(또는 드로어)로 오픈
- 로그인 완료 상태 필수

**3. UI 요소별 기능 정의**

| **요소명** | **유형** | **동작 설명** | **비즈니스 룰** | **에러/예외** |
| --- | --- | --- | --- | --- |
| **알림 리스트** | 표시 (scroll) | 시간 역순으로 알림 나열. 각 알림: 아이콘 + 제목 + 상세 요약 + 시간 | 최대 100건 로드, 스크롤 시 추가 로드 (infinite scroll). 30일 경과 알림 자동 삭제 | 알림 0건 → "새로운 알림이 없습니다" 안내 표시 |
| **진행 중 알림 카드** | 표시 (실시간) | 장시간 Task(대량 등록, 크롤링 등) 시작 시 알림 리스트 **최상단**에 진행 중 카드 표시. **프로그레스 바**  • "완료되면 알려드릴게요" 메시지. 이때 **알림 사운드/푸시 없음** (사용자가 직접 트리거한 작업이므로) | 진행 중 카드는 Task 완료 시까지 상단 고정. 동시 진행 Task가 여러 개면 각각 별도 카드로 표시. 프로그레스 바는 서버 진행률 반영 (폴링 또는 WebSocket) | 서버 연결 끊김 시 프로그레스 바 멈춤 + "연결 확인 중..." 표시. 재연결 시 자동 갱신 |
| **완료 알림 카드** | 표시 + 알림 | Task 완료 시 **알림 사운드/푸시 발생**  • 진행 중 카드가 완료 카드로 전환. **완료 결과 요약 정보** 함께 표시 (예: "상품 등록 완료 — 성공 8건 · 실패 2건") | 결과 요약 형식: 작업 유형별 — 등록: 성공/실패 건수 / 크롤링: 수집 건수 / 싱크: 변동 건수. 실패 건이 1건 이상이면 카드 색상 주황색 강조 | 결과 데이터 로드 실패 시 "작업이 완료되었습니다. 상세 결과는 해당 화면에서 확인해주세요" 표시 |
| **알림 유형 아이콘** | 표시 | 알림 종류별 아이콘 구분: 진행중(⏳) · 수집완료(🔍) · 등록성공(✅) · 등록실패(❌) · 변동감지(🔄) · 역마진(🚨) · 품절(⚠️) | 아이콘+색상으로 긴급도 시각화: 역마진/품절 = 빨간색, 등록실패 = 주황색, 진행중 = 파란색, 나머지 = 기본색 | — |
| **읽음/안읽음 표시** | 표시 | 안읽음 알림은 **배경색 하이라이트**  • 좌측 파란 도트. 클릭 시 읽음으로 전환 | 읽음 전환은 클릭 시점 — 패널 오픈만으로는 전체 읽음 처리 안 됨 | — |
| **해당 화면 바로가기** | 버튼 (알림별) | 알림 클릭 시 읽음 처리 + 해당 화면으로 자동 이동 | 알림 유형별 이동 대상: 수집완료 → `SC-04 수집 진행/결과` / 등록성공 → `RG-02a 등록 완료` / 등록실패 → `RG-02b 등록 실패` / 변동감지·역마진·품절 → `PM-03 변동 확인 상품 리스트` | 대상 화면이 삭제된 경우 → "해당 상품을 찾을 수 없습니다" 토스트 |
| **전체 읽음 처리** | 텍스트 버튼 | 패널 상단 "모두 읽음" 클릭 시 모든 알림을 읽음 상태로 전환 | `G-01 GNB` 뼉지 수도 즉시 0으로 갱신 | — |
| **필터 탭** | 탭 버튼 | 알림 유형별 필터링: 전체 / 소싱 / 등록 / 변동 확인 · 역마진 | 기본값: "전체" 탭 활성. 필터 적용 시 해당 유형만 표시 | 필터 결과 0건 → "해당 유형의 알림이 없습니다" 안내 |

**4. 상태 변화**

- **① 진행 중 (in_progress)**: 사용자가 장시간 Task를 트리거(대량 등록 버튼 클릭 등) → 알림 DB에 레코드 생성 (status: "in_progress"). 프로그레스 바와 함께 표시. **알림 사운드/푸시 없음**
- **② 완료 (completed → unread)**: Task 완료 시 status → "unread"로 전환 + **알림 사운드/푸시 발생** + 결과 요약 데이터 기록 (성공/실패 건수 등)
- **③ 읽음 (read)**: 완료 알림 클릭 시 status → "read"로 변경
- **④ 만료**: 30일 경과 시 자동 삭제 (cron job). 진행 중 상태는 삭제 대상 제외

**5. API/데이터 의존성**

- 내부 알림 DB (CRUD)
- 수집 엔진 → 알림 생성 Webhook (`SC-04 수집 진행/결과` 완료 시)
- 등록 API → 알림 생성 Webhook (Qoo10 등록 완료/실패 시)
- 변동 확인 스케줄러 → 알림 생성 Webhook (변동/역마진/품절 감지 시)
- 알림 설정 API (`ST-04 알림 설정`의 On/Off 설정 반영)

**6. 이탈 경로**

- `SC-04 수집 진행/결과`: 수집 완료 알림 클릭
- `RG-02a 등록 완료`: 등록 성공 알림 클릭
- `RG-02b 등록 실패`: 등록 실패 알림 클릭
- `PM-03 변동 확인 상품 리스트`: 변동 확인 관련 알림 클릭
- `G-01 GNB`: 패널 닫기 또는 패널 외부 클릭

### 1. 온보딩/초기 셋팅

#### [S-01] 로그인/회원가입

**1. 화면 목적**

서비스 진입점. 신규 셀러의 가입과 기존 셀러의 로그인을 처리하여 인증된 세션을 생성한다.

**2. 진입 조건**

- 비로그인 상태에서 서비스 URL 접속 시 자동 노출
- 로그인 상태에서 접근 시 `SC-01 소싱 홈`으로 자동 리다이렉트 (온보딩 완료 시)

**3. UI 요소별 기능 정의**

| **요소명** | **유형** | **동작 설명** | **비즈니스 룰** | **에러/예외** |

**4. 상태 변화**

- **대기 (queued)**: SC-03에서 생성 직후
- **진행중 (running)**: RPA 엔진이 작업 시작. 수집 건수 실시간 증가

**5. API/데이터 의존성**

- 크롤링 Job DB (상태·결과 조회)
- 내부 소싱 상품 DB (수집 결과 저장)

### 보충 4. 공통 폼 입력 패턴

<aside>
🔔

**합의 수준**: PM·Engineer 합의 (추가 도출)

**필요 이유**: 화면별 기능 명세서에 입력 필드가 많은 화면이 다수 존재(OB-05 API Key 입력, SC-01 URL 입력, ED-01 가격·마진율 편집, ST-01 연동 설정 등). 입력 필드의 유효성 검증 방식, 필수/선택 구분, 에러 표시 위치 등의 공통 규칙이 없으면 AI가 화면마다 다르게 구현함.

**연계 문서**: UX 설계 기준 › 핵심 설계 원칙 4번(에러 발생 시 해결책 제시) + 라이팅 원칙 5번(에러는 해결책과 함께)

</aside>

| **항목** | **규칙** | **비고** |

| **실시간 검증 (형식)** | URL 형식, API Key 형식, 이메일 형식 등 클라이언트 측 검증은 입력 즉시 처리 | 키 입력마다 or 포커스 아웃 시 검증 |
